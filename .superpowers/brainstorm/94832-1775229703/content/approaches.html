Design Approaches: Works + Portfolio + Access Control

Three approaches to integrating work creation into Portfolio, ownership tracking into Registry, and dual-layer access control

A

Portfolio-First (Recommended)

Portfolio is the creation hub. Registry is the tracking hub.

  • Portfolio page: Create projects, add works (hybrid batch + one-by-one), link audio files, manage project files, configure project roles via a Settings tab
  • Registry page: Read-only dashboard of "everything I'm involved in" — my works, my stakes, pending reviews, ownership percentages across all projects. Click through to WorkDetail for actions.
  • Project Settings tab: Manage members (add/remove, set roles with color-coded badges), project-level permissions. Small tab icon on each project card.
  • WorkDetail page: Unchanged — confirm/dispute stakes, manage licensing, agreements, export proof
  • File dedup: SHA-256 hash on upload, prompt if duplicate detected

Pros

  • Clear mental model: Portfolio = create & organize, Registry = track & verify
  • Minimal changes to existing Registry flow
  • Natural workflow: you're in your project, you add works

Cons

  • Registry becomes more passive — some users may expect to create works there too
B

Dual Entry Points

Create works from Portfolio OR Registry. Both are active hubs.

  • Portfolio page: Same as A — project-centric work creation, audio linking, project roles
  • Registry page: Keeps existing "Create Work" dialog AND adds ownership dashboard. Users can create works from either location.
  • Shared components: Work creation form is a reusable component used by both pages
  • Project Settings: Same as A — tab on project card

Pros

  • No breaking change — existing Registry create flow stays
  • Flexible: users choose their preferred entry point

Cons

  • Two places to create works = potential confusion
  • More UI surface to maintain
  • Harder to keep consistent
C

Unified Project View

New dedicated Project Detail page becomes the central hub for everything.

  • New Project Detail page: Tabbed view — Works, Files, Audio, Members, Settings. All work creation and management happens here.
  • Portfolio page: Simplified — just lists projects as cards with quick stats. Click to open Project Detail.
  • Registry page: Pure dashboard — ownership tracking, pending reviews, cross-project view of all involvement
  • Project Settings: Full tab within Project Detail (not just an icon)

Pros

  • Cleanest separation of concerns
  • Project Detail page is self-contained — everything about a project in one place
  • Portfolio stays lightweight and scannable

Cons

  • New page to build from scratch
  • More navigation depth (Portfolio → Project Detail → Work Detail)
  • Bigger lift

My Recommendation: Approach A

Portfolio-First gives you the clearest mental model with the least disruption. The Portfolio is where you organize and create (your private space), the Registry is where you track shared ownership (your public/collaborative space). It also aligns with your vision of artist profiles being private — the Portfolio is personal, the Registry is shared. Approach C is the cleanest architecturally but is a bigger build. Approach B adds flexibility but risks confusion.
